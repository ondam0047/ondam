# 기록지 양식 — 사용자 업로드/자동매핑 설계 문서

> 작성: 2026-06 베타 중. 지자체·센터마다 발달바우처 제공기록지 양식이 달라서,
> **치료사가 자기 기록지를 올리면 그 서식으로 출력**되게 만드는 기능의 설계·근거·계획.

---

## 0. 한 줄 요약

빈 발달바우처 기록지(.hwpx)를 **표준 라벨 + 표 기하학**으로 자동 매핑(좌표 추론)하고,
**치료사 본인이 미리보기로 확인·수정**한 뒤 자기 센터 양식으로 저장해 쓴다.
운영자 개입 없음(수백 명 수동 불가). 자동매핑 PoC는 우리 6종에서 **100% 적중** 검증됨.

---

## 1. 배경 / 문제

- 발달재활서비스 **제공기록지**는 보건복지부 표준서식(서식 11호) 기반인데, **지자체·기관마다 레이아웃이 다름**.
- 베타 사용자들이 "우리 양식 그대로 쓰고 싶다"고 요청.
- 지금은 양식마다 **개발자가 코드에 좌표를 박고 배포**해야 함 → 수백 센터로 확장 불가.

### 결정된 정책
- **센터당 1양식** (다지자체 센터는 예외로 두고 베타에선 미지원).
- **운영자 관리 없음** → 치료사가 **셀프로 업로드·확인·수정**.
- **회기 칸은 모든 양식 5칸 고정**, 6회기부터 ZIP 분할(기존 정책 유지).
- 자동 채움 대상은 **편집가능 .hwpx(표 있는 빈 양식)만**. .hwp·스캔·PDF는 미지원 안내.

## 2. 핵심 통찰 (왜 이게 가능한가)

1. **규제 닫힌 스키마** — 임의 문서가 아니라 표준서식의 변형. 필드 집합과 라벨 어휘가
   닫혀 있음 → open-domain 문서이해가 아니라 **닫힌 스키마 라벨 매칭**. ML 불필요.
2. **HWPX는 완전 구조화 XML** — `<hp:tc colAddr rowAddr colSpan rowSpan>` + 라벨 텍스트.
   OCR·렌더링 없이 표 격자와 라벨을 그대로 읽음.
3. **기하 규칙이 일관** — 헤더는 "라벨 오른쪽 칸", 회기는 "월일/내용 헤더행의 뒤 칸들이
   날짜 축, 그 축 열을 각 항목 줄이 공유". → **결정론적 규칙으로 좌표 도출 가능**.

→ 그래서 **라벨·기하 결정론 리졸버**가 정답. 비주얼 매퍼(HWPX 렌더 어려움)와 ML(과잉)은 배제.

## 4. 자동매핑 리졸버 (PoC 검증 완료 ✅)

- 파일: `scripts/resolver-poc.ts` (실행 시 6종 대조 적중률 출력).
- 결과: **378/378 = 100%** (play 54, dongtan 59, namyangju 54, suncheon 49, wonju 69, daegu 93).
- 실행:
  ```
  # @/ 별칭 resolver 훅 필요 (아래 7장 참고)
  node --experimental-strip-types --loader /tmp/alias-hook.mjs scripts/resolver-poc.ts
  ```

### 4.1 필드 온톨로지 (닫힌 목록)
기관명, 이용자 이름, 생년월일, 제공영역 / 회기 날짜, 시작시간, 종료시간, 담당재활사 /
바우처(분), 추가구매(분) / 총금액, 바우처금액, 자부담금액 /
승인일자, 승인번호, 이용자상태, 서비스결과, 비고·특이사항.

### 4.2 리졸버 알고리즘 (결정론)
1. HWPX → 표 격자 모델(셀: row/col/colSpan/text/norm/단락수). `"/" ":" 공백`은 빈칸 취급.
2. **안내문 셀 제외**: `☞`, "표기합니다/바랍니다/기재합니다/받아야", 길이>24 → 라벨로 안 봄.
   (이거 없으면 "…담당자 **성명** 표기"가 이름 라벨로 오인됨)
3. **헤더**: 라벨(제공기관명/성명/생년월일/제공영역) → **같은 행 오른쪽 인접 셀** = 값.
   (빈칸 여부 무관 — 잔여 데이터 있어도 위치는 같음)
4. **회기 축**: norm에 **"월일" AND "내용" 둘 다** 든 헤더 행을 찾음(생'년월일' 오검 방지).
   라벨 span 뒤 칸들 = 날짜 열. `누계/합계/소계/계` 제외, 앞에서 **5개**만.
5. **회기 항목 줄**: 시작시간/종료시간/바우처/추가구매 라벨 행 × 날짜 축 열.
   - 바우처(분)은 '추가구매' 행 바로 위 '바우처' 행으로 구분(금액쪽 바우처와 분리).
6. **금액**: '총이용금액' 섹션. 그 아래 '자부담'+'총금액' 라벨이 있으면 **3행 분해형**
   (바우처금액/자부담/총금액). 없으면 단일 총금액.
7. **결과표**: 메인 표 외에서 결과 헤더 라벨(승인일자/승인번호/제공일자/이용자상태/
   서비스결과/기타사항/시간) ≥2개인 행 = 헤더. 그 아래 빈 행 = 회기(5개). 열은 라벨로 매핑.
   - **시각 단어 "시간"은 정확히 일치만**(제공/시작/종료시간 오검 방지).
   - **승인일자 칸 단락이 2개면 날짜(p0)+시각(p1)로 분리** (익산형).
   - '이용자의상태및서비스결과' 합쳐진 칸 → 'result' 하나로.
8. **다서비스 블록**: '시작시간' 라벨이 여러 개면 각 블록의 c0 서비스명(언어/미술/음악/기타)로
   `serviceBlocks` 구성, 4번째는 default(기타). 출력 시 치료사 serviceType로 블록 선택.
9. **비고/특이사항** → 값 칸.

### 4.3 신뢰도/커버리지
리졸버는 "어떤 필드를 찾았는지"를 알므로, **필수 필드 누락 여부 = 커버리지 리포트**.
이게 그대로 미리보기 화면의 ⚠️ 표시·확인 게이트를 구동.

## 5. 아키텍처 (본 구현 시 만들 것)

### 5.1 데이터 모델 (제안)
```
Form {
  id
  ownerCenterId      // 센터당 1양식
  name               // 표시명 (예: "우리 센터 양식")
  templateBlob       // 업로드한 빈 .hwpx (object storage 또는 DB blob)
  structureHash      // 파싱 구조 지문 — 동일 구조면 스펙 재사용
  spec(JSON)         // 좌표맵 (현 CoordSpec 일반화). 리졸버 출력 + 사용자 보정 결과
  status             // draft | confirmed
  createdBy, version, createdAt
}
Center.recordFormId  // (현 recordForm 문자열 키를 FK로 일반화)
```
- 해석 순서: **우리 센터 양식 > 표준형 폴백**. (지자체 공유 계층은 베타에선 생략 가능)
- 엔진(`record-hwpx.ts`)은 코드 상수 대신 **DB의 spec(JSON)을 읽도록** 일반화.
- `structureHash`: 같은 구조(예: 같은 지자체 양식)를 다른 센터가 올리면 **이미 푼 스펙 재사용**.

### 5.2 CoordSpec(JSON) 형태 — 이미 있는 것 그대로
`lib/record-hwpx.ts`의 `CoordSpec` 타입을 JSON으로:
헤더(org/name/birth/serviceArea), date/start/end/voucher/extra/amount(+voucherAmount/copayAmount),
result[] (date/time/apprDate/apprNum/status/result), note, serviceBlocks/serviceBlockDefault, detail(별지).
좌표는 `[table, row, col, p?]`.

## 6. 셀프 보정 UX (운영자 없음)

업로드 → **확인/수정 화면** → 저장 흐름.

1. **자동 매핑**(리졸버) — 표준 양식이면 대부분 칸이 이미 잡힘.
2. **양식을 HTML 표로 재구성**(셀 병합·텍스트 그대로) + 인식된 칸에 **색 뱃지**(역할명).
3. **"샘플로 채워보기"** 토글 → 더미 데이터가 실제처럼 들어간 모습 확인(= 안전망).
   - 또는 "한글로 받아 확인" (실제 .hwpx 다운로드).
4. **틀린 칸 = 클릭 → 역할 드롭다운**(4.1 닫힌 목록 + "비우기").
5. **못 찾은 칸**은 ⚠️ 체크리스트 → 클릭해 지정.
6. **저장** → 우리 센터 양식 확정.

### 보정 단위 (덜 헷갈리게 — 구조에 맞춤)
- 헤더(이름·생년월일·기관명): **칸 클릭** 태깅.
- 회기 부분: **"이 줄이 시작시간"** 식 **줄 태깅**, 날짜 열은 자동 정렬.
- 결과표: **열 헤더** 태깅.
- 보통 틀려도 클릭 2~3번. (자동매핑 90%+)

## 7. 안전 가드레일

- **확인 전엔 실제 기록지에 자동 사용 금지** — 미리보기/샘플채움이 유일·필수 게이트.
- 못 채운 칸은 **틀린 값 대신 빈칸 + 경고**(정산 사고 방지).
- .hwp/스캔/PDF·표 없는 양식 → 업로드 단계에서 **"자동 채움 불가" 안내**.
- 잘못 태깅 책임은 치료사 본인(어차피 제출 책임자). 그래서 미리보기를 강하게.

## 8. 빌드 순서 (제안)

1. **양식 DB화 + 업로드** — Form 모델, 템플릿 blob 저장, `Center.recordFormId` FK화,
   엔진이 DB spec(JSON) 읽게 일반화.
2. **리졸버 연결** — `scripts/resolver-poc.ts` 로직을 `lib/`로 승격, 업로드 시 자동 spec 제안.
3. **HTML 표 재구성 + 샘플채움 미리보기** ← 안전망 먼저.
4. **셀프 보정 UI**(줄/칸 태깅).
5. **내 설정 → "우리 센터 양식 올리기"** 진입점.
6. `structureHash` 재사용(선택, 나중).

> 추천: **3(미리보기)부터** 만들어 안전망을 먼저 세우고 4(보정)를 얹기.

## 9. 열린 이슈 / 나중에 결정

- 다지자체 센터(한 센터가 여러 지자체 아동) → 베타 미지원. 추후 아동/서비스별 양식 해석 필요?
- 비표준 라벨("성명"→"아동명") 동의어 사전 확장 범위.
- 담당재활사 이름·기타재활 블록의 서비스명 빈칸 자동 채움 여부.
- 원주 금액 분배가 "분 비례"가 맞는지(정률/정액 본인부담 가능성) — 지역 확인 필요.
- 동탄·순천·원주의 '이용자 상태' 칸은 현재 빈칸(우리 데이터는 결과 1칸).

## 10. 기술 참고

- **@/ 별칭 + TS 직접 실행** (스크립트 테스트용): node 가 tsconfig paths 를 안 읽으므로 loader 훅 필요.
  `/tmp/alias-hook.mjs`:
  ```js
  import { pathToFileURL } from 'node:url';
  import { resolve as resolvePath } from 'node:path';
  import { existsSync } from 'node:fs';
  export async function resolve(spec, ctx, next) {
    if (spec.startsWith('@/')) {
      let abs = resolvePath(process.cwd(), spec.slice(2));
      if (!/\.[a-z]+$/.test(abs)) { if (existsSync(abs+'.ts')) abs+='.ts'; else if (existsSync(abs+'.tsx')) abs+='.tsx'; }
      return next(pathToFileURL(abs).href, ctx);
    }
    return next(spec, ctx);
  }
  ```
  실행: `node --experimental-strip-types --loader /tmp/alias-hook.mjs <파일>`
- **HWPX 파싱 헬퍼**: `lib/hwpx.ts` (`readSection0`, `patchSection0`, `parseZipEntries`, `buildZip`).
- **채움 엔진**: `lib/record-fill.ts` (`fillCells(xml, edits[])`).
- **셀 좌표**: 표는 문서 순서 인덱스, 셀은 `<hp:cellAddr colAddr rowAddr>`, 단락은 셀 내 `<hp:p>` 순번.
- **빈 셀 채우기**: 빈 셀은 `<hp:run charPrIDRef=".."/>`(자가닫힘) → `<hp:run ..><hp:t>값</hp:t></hp:run>`,
  `<hp:linesegarray>` 제거해 한글이 재계산하게 함.
